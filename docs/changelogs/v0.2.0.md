# v0.2.0(2026-07-08)

## 概述

修复 v0.1 检索的架构缺陷:候选池从"向量 top-50 独裁"改为"向量 ∪ FTS5 关键词"双通道并集,精确关键词命中者不再因语义分平庸而丧失参赛资格。附带补齐去噪清单、增加两个来源过滤参数。全量重建 13,226 块/156s,五连测回归全绿且两问分数上涨。

## 已确认的机制(含证伪记录)

- **两阶段召回的结构性缺陷(主根因)**:HTAdsCore 40006 定案块措辞("排查定案")与查询("根因")不同构,cos 仅 0.47 进不了向量 top-50,BM25 只在池内重排——关键词再强也无参赛资格。修复后该块模糊查询第 7、消歧查询 top-1。
- **"清洗提升向量分"假设被部分证伪**:去掉 task-notification 垃圾仅使 cos 0.430→0.472,远不足以进池。垃圾污染是真问题但非主根因——这次证伪直接把方案主线从"修数据"扭到"修架构"。
- **验收标准本身被修订**:"模糊查询下特定块必须 top-3"是错误标准——40006 出现在 5+ 个真实项目上下文,多个真实相关块并列是正确行为;盯单块调参 = teaching to the test,拒绝。
- **按已知标记精准去噪,不见尖括号就杀**:全量审计证实对话正文大量存在 `Set<string>`、`<uid>` 等代码占位符,黑名单只收系统注入标记。
- **新发现(遗 v0.3)**:BM25 对泛词("报错""根因")与精确标识符("40006")同权,存在不含关键标识符却满分的块;需停用词/加权。

## 改动列表

- extractor:去噪黑名单增补 `<task-notification>`(含 result/usage 全家族)、`<bash-input/stdout/stderr>`(共滤除 67 条系统消息、40 块噪音)
- store:新增 FTS5 虚表(jieba 分词入库,与查询同一分词器);`SCHEMA_VERSION=2` 触发自动重建;`load_matrix` 行首增加块 id;新增 `fts_candidates()`
- searcher:候选池改为向量 top-50 ∪ FTS5 top-50 并集,统一 cos+BM25 融合重排(权重不变 6:4)
- cli:search 新增 `--before YYYY-MM-DD`、`--exclude-project NAME`(可重复、可组合),行与矩阵同步裁剪
- tokenizer:分词抽为独立模块,入库/查询共用
