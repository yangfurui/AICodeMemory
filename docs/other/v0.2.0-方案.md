# claudeMemory v0.2.0 方案

> 2026-07-08 起草。主线:给召回引入关键词独立通道与来源先验。动手前经确认。

## 调查结论(v0.2 的依据,过程见 v0.1.0-方案"已知课题")

以"40006 报错的根因是什么"未召回 HTAdsCore 定案块(537e1abd 块 51)为样本,逐一假设验证:

1. **主根因(架构缺陷)**:完美答案块在库里,cos 仅 0.43(清洗后 0.472),进不了向量 top-50 候选池;而 BM25 只在池内重排——**精确关键词命中者若向量落榜,彻底没有参赛资格**。深层原因:bge-small 对"定案↔根因"这类同义不同词的对齐能力有限,查询措辞与目标块措辞不同构时 cos 必然平庸,恰恰这种时刻最需要关键词通道兜底。
2. **次因(去噪缺口,已实测)**:`<task-notification>` 等系统注入的伪 user 消息未被过滤,UUID 垃圾稀释向量(污染版 0.430 vs 清洗版 0.472)。
3. **误报排除**:quliu-train/eac126cd"0 块含 40006"是正确行为——其 40006 全在 tool_use/tool_result 里,本就该滤。**去噪策略本身不动**,不保留工具输出。
4. **自指污染(v0.1 已知课题)**:当天引用 40006 当例子的会话措辞与查询同构,天然占 top-1。

## 技术方案

### A. 混合召回(修主根因)

候选池从"向量 top-50"改为**"向量 top-50 ∪ 关键词召回 top-50"取并集**,再统一做 cos+BM25 融合重排(权重不变 6:4,BM25 在并集内归一;FTS5 独家候选补算单条 cos,开销可忽略)。

关键词召回用 **SQLite FTS5**(内置全文索引,延续零依赖路线):chunks 入库时同步写一张 FTS5 虚表,存 jieba 分词后空格拼接的文本;查询时同样分词后 MATCH。

### B. 去噪补漏(修次因)

- `_COMMAND_MARKERS` 增补 `<task-notification>`;
- 入库前全量审计一遍高频 XML 标签(一次性脚本扫源数据统计,按结果决定清单,防再漏);
- 去噪/切块逻辑变更 → bump `SCHEMA_VERSION` → 库自动作废全量重建(既有机制,重建约 5 分钟)。

### C. 来源先验(修自指污染,取最小方案)

v0.2 只做两个显式检索参数,不做隐式加权(可控、零回退风险):

- `cmem search --before 2026-07-08`:只检索该日期前的块
- `cmem search --exclude-project claudeMemory`:排除指定项目

时间衰减/引用者降权这类隐式先验,证据不足,留 v0.3 做实验(需要先攒评估集)。

## 验收标准(实测后修订,见下方说明)

1. ~~"40006 报错的根因是什么" → 定案块进 top-3~~ **修订**:该标准设计有误——"40006"在数据里出现于 5+ 个真实项目上下文,模糊查询下多个真实相关块并列是正确行为,盯着单块调参属于 teaching to the test(MemPalace 踩过的坑),拒绝。修订后标准:(a) 定案块获得候选资格(v0.1 连 top-50 都进不了)→ **实测第 7**;(b) 消歧查询"HTAdsCore 40006 缺聚合模块"→ **实测 top-1(0.758)**;(c) 参数组合下 top-1 → **实测达成**
2. v0.1 五连测不回退 → **全部保持 top-1,两问分数上涨**(安静期 0.802→0.834、瘦身 0.694→0.728,FTS 通道加成)
3. 重建后同量级 → **13,226 块(-40,恰为滤掉的噪音),156s**
4. 查询延迟仍亚秒级 → **达成**
