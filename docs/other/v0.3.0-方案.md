# AICodeMemory v0.3.0 方案 — 数据安全版(档案化)

> 2026-07-08 起草。用户裁决:"原始数据肯定是要保留的,不管它放在哪里"。
> 原定 v0.3(MCP 集成)顺延为 v0.4——数据安全优先于功能。

## 定位修正(本版的根本动因)

v0.1/v0.2 把库定义为"派生缓存,可随时从源重建,不用备份"。**该定位对普通用户是错的**:源(`~/.claude/projects/`)被 Claude Code 按 30 天滚动清理,库中超过 30 天的内容是全世界唯一副本——**库不是缓存,是档案**。

由此暴露的雷:`store._check_versions` 在 schema/模型变更时 `DELETE FROM chunks` 全清重灌,而重灌只能覆盖存活的源 → **用户每次升级,30 天前的记忆被销毁**(v0.2 升级已发生一次,未造成损失纯因用户源均在窗口内)。

## 数据模型:三层生命周期

| 层 | 内容 | 生命周期 | 再生性 |
|---|---|---|---|
| **raw 原始层**(新增) | 源 jsonl 的 gzip 拷贝,`~/.cmem/raw/<项目>/<会话>.jsonl.gz` | **永不自动删除** | 不可再生(底片) |
| **text 档案层** | chunks.text 对话原文 | **永不自动删除** | 可从 raw 重新提取 |
| **索引层** | embedding、chunks_fts | 随时可弃 | 可从 text 重算 |

## 技术方案

### A. raw 原始层

- `cmem index` 处理每个"新/有变化"的会话时,顺手把源 jsonl gzip 压缩存入 raw 层(mtime 变化则覆盖更新——与索引同一套增量判定,零额外扫描);
- 源文件从磁盘消失 ≠ 删除其 raw 存档(append-only);
- 空间预估:实测源 241MB/月,jsonl 结构重复度高,gzip 后约 30-50MB/月,一年 <600MB,可接受;
- raw 层的价值:任何未来的去噪/切块算法改进,都能回溯应用到全部历史(含已过期源)。

### B. 拆雷:版本轴拆分,升级永不丢档案

把单一 `SCHEMA_VERSION` 拆成两根独立版本轴:

- **`EXTRACT_VERSION`**(去噪/切块算法)变更 → 从 **raw 层**重新提取 text(raw 是源的超集,覆盖已过期历史)→ 重算索引;
- **`MODEL`**(embedding 模型)变更 → text 一字不动,仅从库内 text 重算向量与 FTS;
- 纯表结构变更 → SQL 迁移,不触碰数据。

任何路径下,**text 与 raw 永不删除**。

### C. 契约固化

两条不变量写成测试,防未来改动违约:

1. 源文件消失后运行 `cmem index` → 该会话的块与 raw 存档完整保留、可检索;
2. 模拟模型/算法版本变更 → 升级后 text 行数不减、raw 文件不减。

### D. 体检与备份指引

- `cmem status` 增加:raw 层规模(文件数/大小)、`PRAGMA integrity_check` 结果;
- README 定位段与备份建议反转:**`~/.cmem/` 整个目录(库+raw)包含超过 30 天窗口的唯一记录,应纳入用户备份**(Time Machine/云盘);产品叙事改为"Claude Code 只给你 30 天,AICodeMemory 把它变成永久"。

## 验收标准

1. **遗忘对抗**:把某个已索引会话的源文件移走(模拟 30 天清理)→ `cmem index` → 块仍在、检索正常、raw 存档仍在;
2. **升级不丢档**:模拟 MODEL 变更触发重算 → text/raw 行数与变更前一致,检索正常;模拟 EXTRACT_VERSION 变更 → 从 raw 重提取成功(含已"消失"源的会话);
3. **raw 增量**:二次 index 秒级跳过未变化会话的存档;会话有新内容时存档被更新;
4. **回归**:五连测不回退;status 显示 raw 规模与 integrity ok。
